## CRITICAL (must fix before production)

### C-1. XSS via unsanitized `content_html` in e-learning chapter rendering

**File:** `src/app/(dashboard)/learner/courses/[courseId]/page.tsx`, line 841  
**Description:** `ch.content_html` from the database (generated by OpenAI) is injected via `dangerouslySetInnerHTML` with **no DOMPurify sanitization**. The content is AI-generated and stored in DB, but an admin with DB write access (or an OpenAI prompt injection) could inject arbitrary JS.  
**Other files sanitize correctly:** emails page uses `DOMPurify.sanitize()`, documents page uses `sanitizeHtml()`, signatures use `sanitizeSignatureSvg()`. This is the only `dangerouslySetInnerHTML` without sanitization of DB-sourced HTML.  
**Fix:** Wrap with `DOMPurify.sanitize(ch.content_html)` before rendering. Import DOMPurify as done in other pages.

### C-2. `renderMarkdown()` in programs page is NOT a sanitizer

**File:** `src/app/(dashboard)/admin/programs/[id]/page.tsx`, lines 59-69, used at lines 780, 790, 803  
**Description:** `renderMarkdown()` does basic regex-based markdown-to-HTML conversion. While it escapes `<`, `>`, and `&` before transforming, the function inserts `<strong>`, `<em>`, `<li>`, `<ul>`, `<br>` tags. If `program.description` or `program.objectives` contain crafted text that exploits the regex order (e.g., bold markers wrapping injection payloads), XSS is possible. Unlike the documents page which uses DOMPurify, this does not use a proper sanitizer.  
**Fix:** Pass the output through `DOMPurify.sanitize()` before `dangerouslySetInnerHTML`.

### C-3. CommandPalette queries with NO entity_id filter

**File:** `src/components/layout/CommandPalette.tsx`, lines 48-51  
**Description:** All four queries (`sessions`, `clients`, `learners`, `trainers`) use user-typed search text in `.ilike()` / `.or()` without ANY `.eq("entity_id", ...)` filter. While RLS on the server client filters by entity, this violates the CLAUDE.md rule #2 absolute requirement and represents defense-in-depth failure. If RLS is misconfigured or bypassed (service role leak), this exposes cross-tenant data.  
**Fix:** Add `.eq("entity_id", entityId)` to all four queries. Get `entityId` from the EntityContext.

### C-4. `.or()` PostgREST filter injection via unsanitized user input

**Files:**
- `src/app/(dashboard)/admin/clients/apprenants/page.tsx`, line 63
- `src/app/(dashboard)/admin/trainers/liste/page.tsx`, line 55
- `src/app/(dashboard)/admin/clients/apprenants/liste/page.tsx`, line 108
- `src/app/(dashboard)/admin/activity/page.tsx`, line 174
- `src/components/layout/CommandPalette.tsx`, lines 50-51

**Description:** User search input is interpolated directly into PostgREST `.or()` filter strings: `.or(\`first_name.ilike.%${search.trim()}%,...\`)`. PostgREST `.or()` DSL uses commas and dots as delimiters. A malicious user could inject additional filter clauses (e.g., searching for `%,id.neq.`) to manipulate the query. Note: `src/app/(dashboard)/admin/crm/prospects/liste/page.tsx` line 57 documents this risk and strips dangerous characters -- but the other files do NOT.  
**Fix:** Sanitize user input before interpolation into `.or()` strings. Strip characters: `,`, `.`, `(`, `)`, `!`. Or use the Supabase `.ilike()` method instead of `.or()` with string interpolation.

## HIGH (fix soon)

### H-1. track-view endpoint accepts entity_id from request body (trust boundary)

**File:** `src/app/api/documents/track-view/route.ts`, lines 20, 39, 66  
**Description:** This public (no-auth) endpoint accepts `entity_id` from the JSON body and inserts it into `document_views`. While there is a validation check at line 39-49 that verifies the document belongs to the declared entity, the entity_id itself still comes from the client. If `entity_id` is omitted (line 39: `if (entity_id)`), the validation is skipped entirely and the insert proceeds with `entity_id: null`.  
**Fix:** Make `entity_id` required. Always derive entity_id from the document record, never trust the client.

### H-2. Missing entity_id on service-role cron queries (cross-tenant data exposure)

**Files:**
- `src/app/api/formations/automation-rules/run-cron/route.ts`, line 285: `supabase.from("entities").select("id, name")` -- fetches ALL entities (by design for multi-entity cron), but downstream queries at lines 295, 361, 398 must each filter by entity. Lines 62-65 (rule-scoped mode) fetch `formation_automation_rules` by ID without entity_id filter.
- `src/app/api/crm/quotes/process-reminders/route.ts`, lines 42-47: Queries `crm_quotes` with status and date filters but NO entity_id. Processes ALL entities' quotes in one batch.
- `src/app/api/documents/process-sign-reminders/route.ts`, lines 42-47: Same pattern -- queries `documents` without entity_id filter.

**Description:** These cron endpoints use service_role keys (bypasses RLS). Queries fetch data across ALL entities. While this is intentional for cron iteration, it means a logic bug could send one entity's invoice reminder to another entity's client.  
**Fix:** Add explicit entity_id filters to each downstream operation, or partition processing per-entity with clear boundaries.

### H-3. `questionnaire_sessions` table has no entity_id column

**File:** `supabase/schema.sql`, lines 260-267  
**Description:** `questionnaire_sessions` is a join table (questionnaire_id, session_id) with no `entity_id`. RLS policy exists in `add_missing_rls_policies.sql` but relies on chained lookups. Any direct query on this table without joining through `sessions` or `questionnaires` could leak cross-entity associations.  
**Severity note:** Mitigated by RLS joining to entity-filtered parent tables.

### H-4. `formation_absences` and `formation_documents` tables have no entity_id column

**File:** `supabase/migrations/add-formation-tabs-4-5-6.sql`, lines 10-36  
**Description:** These tables reference `session_id` but have no direct `entity_id`. RLS exists but relies on chained lookups through `sessions.entity_id`. If the application queries these tables directly (e.g., in API routes) using service_role, there is no entity_id to filter on.  
**Fix:** Add `entity_id` column for defense-in-depth, or ensure all application-level queries join through sessions.

### H-5. Silent error swallowing in catch blocks

**Files (sample -- 50+ occurrences found):**
- `src/app/(dashboard)/admin/notifications/page.tsx`, lines 77, 99, 113 -- catch { } with no toast
- `src/app/(dashboard)/admin/ma-semaine/page.tsx`, lines 141, 157
- `src/app/(dashboard)/admin/page.tsx`, line 114 -- `catch { /* ignore */ }`
- `src/app/(dashboard)/learner/courses/[courseId]/page.tsx`, line 460 -- `catch { /* ignore */ }`
- `src/app/api/documents/process-sign-reminders/route.ts`, line 132 -- `catch { sent = false; }`

**Description:** Over 50 catch blocks across the application silently swallow errors with no user feedback (no toast), no error tracking, and no telemetry. This violates CLAUDE.md rule #5 ("Jamais d'action async sans try/catch + toast"). Users experience silent failures.  
**Fix:** Add toast.error() or error tracking in each catch block. For API routes, ensure proper error logging.

### H-6. `enrollments_client_read` RLS policy lacks entity_id check

**File:** `supabase/schema.sql`, lines 731-743  
**Description:** The `enrollments_client_read` policy checks `client_id IN (SELECT client_id FROM learners WHERE profile_id = auth.uid())` but does NOT include `AND session_id IN (SELECT id FROM sessions WHERE entity_id = auth.user_entity_id())`. A client user with learners enrolled in multiple entities could see cross-entity enrollment data.  
**Fix:** Add `AND session_id IN (SELECT id FROM sessions WHERE entity_id = auth.user_entity_id())` to the USING clause.

### H-7. Public questionnaire submit uses service_role with no entity scope

**File:** `src/app/api/questionnaire/public-submit/route.ts`  
**Description:** This public (no-auth) endpoint uses `SUPABASE_SERVICE_ROLE_KEY` (line 6-11) to query and insert into multiple tables: `questionnaire_tokens`, `questionnaires`, `questions`, `learners`, `sessions`, `questionnaire_responses`. All queries filter by token/ID but none filter by entity_id. A valid token from entity A allows inserting responses that reference entity A's data, which is correct -- but if token validation has any flaw, cross-entity writes are possible.  
**Fix:** After resolving the token, verify the session's entity_id and pass it through the response insertion.

## MEDIUM (fix when possible)

### M-1. `calendar/export` route queries without explicit entity_id

**File:** `src/app/api/calendar/export/route.ts`, lines 29-50  
**Description:** Queries `learners` by `profile_id` and `enrollments` by `learner_id` without explicit `entity_id` filter. Relies on RLS (uses server client, not service role) but violates the CLAUDE.md absolute rule.  
**Fix:** Add `.eq("entity_id", entityId)` derived from the authenticated user's profile.

### M-2. `emails/history` route missing entity_id on direct query

**File:** `src/app/api/emails/history/route.ts`, line 27  
**Description:** Queries `email_history` with various filters but entity_id compliance should be verified. The route uses `requireRole` (auth.supabase which has RLS), but the CLAUDE.md rule requires explicit filtering.  
**Fix:** Add `.eq("entity_id", auth.profile.entity_id)` to the query.

### M-3. Potential PostgREST filter manipulation in CRM suivi

**File:** `src/app/api/crm/suivi/route.ts`, line 91  
**Description:** `.or()` is called with dynamic values. While this is server-side (less exposed), the pattern should use parameterized filtering.  
**Fix:** Validate input values before interpolation into `.or()` strings.

### M-4. `formation_invoice_lines` table - no direct entity_id

**File:** `supabase/migrations/add_formation_invoice_lines.sql`  
**Description:** `formation_invoice_lines` references `invoice_id` but has no `entity_id`. RLS chains through parent invoice. Service-role queries have no entity boundary.  
**Fix:** Add entity_id column or always join through formation_invoices in application code.

### M-5. `signature_evidence` table - audit data without entity_id

**File:** `supabase/migrations/add_signature_evidence.sql`  
**Description:** `signature_evidence` stores forensic data (IP, user agent) linked to `signature_id`. No entity_id column. While this is audit data, cross-entity queries via service role would mix evidence from different tenants.  
**Fix:** Add entity_id column for partition isolation.

### M-6. Rate limiting is in-memory only

**File:** `src/lib/rate-limit.ts`  
**Description:** The rate limiter uses a JavaScript Map (in-memory). On Netlify Functions (serverless), each function invocation gets a fresh memory space. This means the rate limiter resets on every cold start and provides NO effective protection against brute-force attacks on public endpoints like `/api/emargement/sign`, `/api/documents/sign`, `/api/questionnaire/public-submit`.  
**Fix:** Use Redis-backed rate limiting (e.g., Upstash Redis) or Supabase-backed counters for serverless environments.

## LOW (nice to have)

### L-1. `pappers_cache` table has no entity_id

**File:** `supabase/migrations/add_pappers_cache.sql`  
**Description:** Cache is shared across entities (keyed by SIRET). This is acceptable for a public company data cache but means entity A can see company lookups initiated by entity B.  
**Impact:** Minimal -- Pappers data is public company information.

### L-2. `competency_taxonomy` table is entity-agnostic

**File:** `supabase/migrations/enrich_trainers_cv_library.sql`  
**Description:** Shared taxonomy across entities. By design, but worth documenting the intent.

### L-3. Some catch blocks use console.warn instead of structured logging

**Files:** Multiple in `src/lib/services/batch-email-handler.ts`, `src/lib/services/gamma.ts`  
**Description:** Non-critical operations (extra polls, optional data loading) use `console.warn` in catch blocks. Acceptable for optional operations but should be captured by error tracking in production.

### L-4. `verifyCronAuth` fallback pattern

**Files:** CRM notification routes (`daily-digest`, `weekly-summary`, `generate`)  
**Description:** These routes accept either CRON_SECRET or user session auth. The dual-auth pattern is well-implemented but increases attack surface. If both CRON_SECRET and session auth are attempted, failure of one falls through to the other.  
**Impact:** Minimal -- the pattern is standard for admin-triggerable crons.
