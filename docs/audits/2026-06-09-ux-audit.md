## CRITICAL - Bloque la production

### C-1. Dashboard admin : aucun try/catch sur fetchAll()
**Page** : `src/app/(dashboard)/admin/page.tsx` (L161-178)
**Probleme** : `fetchAll()` appelle 6 fonctions Supabase en `Promise.all()` sans try/catch global.
Si une seule echoue (RLS, reseau, table manquante), l'erreur non capturee crashe la page.
Aucun des sub-fetchers individuels (fetchKPIs, fetchAlerts, etc.) n'a de try/catch non plus.
**Impact** : Page blanche pour l'admin = plateforme inutilisable.
**Fix** : Envelopper `fetchAll()` dans try/catch + toast d'erreur + `setLoading(false)` dans `finally`.

### C-2. Dashboard admin : aucun toast importé pour les erreurs
**Page** : `src/app/(dashboard)/admin/page.tsx`
**Probleme** : `useToast` n'est jamais importé ni utilisé. Aucun feedback utilisateur en cas d'erreur.
**Fix** : Importer `useToast`, ajouter des toasts destructive dans les catch.

### C-3. Apprenants liste : `confirm()` natif pour suppression
**Page** : `src/app/(dashboard)/admin/clients/apprenants/liste/page.tsx` (L140)
**Probleme** : `handleDelete` utilise `window.confirm()` au lieu d'un Dialog shadcn/ui.
Pas de feedback si la suppression echoue, pas de spinner de loading pendant la suppression.
La suppression ne verifie pas si l'apprenant a des enrollments existants.
**Impact** : Suppression accidentelle en 1 clic; UX non-professionnelle pour B2B.
**Concerne aussi** : 14 autres pages utilisent `confirm()` (incidents, lieux, users, financeurs,
amelioration, automation, crm/prospects, crm/formulaires, crm/automations, elearning, trainers/liste).
**Fix** : Remplacer tous les `confirm()` par des `Dialog` + `DialogContent` avec bouton destructive.

### C-4. Apprenants liste : raw HTML partout au lieu de shadcn/ui
**Page** : `src/app/(dashboard)/admin/clients/apprenants/liste/page.tsx`
**Probleme** : Utilise `<input>`, `<button>`, `<select>` natifs + styles inline au lieu de
`Input`, `Button`, `Select` de shadcn/ui. Meme chose pour la pagination (boutons raw HTML).
Incoherent avec le reste de la plateforme.
**Concerne aussi** : `reports/factures/page.tsx` (L220-256 : raw `<select>`, `<input>`, `<button>`),
`reports/incidents/page.tsx`, `admin/lieux/page.tsx`, `admin/users/page.tsx`.
**Fix** : Migrer vers les composants shadcn/ui pour la coherence visuelle.

## HIGH - Mauvaise experience utilisateur

### H-1. Apprenants liste : envoi d'emails en boucle sequentielle sans rate limiting
**Page** : `src/app/(dashboard)/admin/clients/apprenants/liste/page.tsx` (L175-201)
**Probleme** : `handleMassEmail()` envoie les emails un par un dans une boucle `for..of` avec
`fetch("/api/emails/send")`. Aucun rate limiting, aucune barre de progression, les erreurs
individuelles sont silencieusement avalees (`catch { /* continue */ }`).
**Impact** : Sur 100 apprenants, l'UI est bloquee pendant des minutes sans feedback.
**Fix** : Utiliser un batch endpoint ou Promise.allSettled avec concurrence limitee + progress bar.

### H-2. Apprenants cards : pagination client-side apres fetch server
**Page** : `src/app/(dashboard)/admin/clients/apprenants/page.tsx`
**Probleme** : Le filtre par entreprise (`companyFilter`) est applique cote client sur les resultats
deja pagines cote serveur. Si un apprenant d'une entreprise est sur la page 2, il sera invisible
quand on filtre par cette entreprise depuis la page 1.
**Fix** : Appliquer le filtre entreprise dans la query Supabase (.eq('client_id', ...)).

### H-3. Dashboard admin : aucun etat d'erreur visible
**Page** : `src/app/(dashboard)/admin/page.tsx`
**Probleme** : En plus de l'absence de try/catch (C-1), le composant ne gere aucun etat d'erreur.
Si les KPIs echouent, l'admin voit des 0 partout, ce qui est trompeur.
Les widgets Alertes et Taches en retard ne s'affichent que s'il y a des donnees, mais
s'il y a eu une erreur de fetch, on ne peut pas distinguer "tout va bien" de "echec silencieux".
**Fix** : Ajouter un etat `error` avec un composant Alert shadcn/ui visible.

### H-4. Factures : pas de skeleton/loading correct
**Page** : `src/app/(dashboard)/admin/reports/factures/page.tsx`
**Probleme** : Utilise `SkeletonTable` et `SkeletonStats` (imports corrects) mais les filtres
sont du raw HTML (`<select>`, `<input>`, `<button>`), creant une incoherence visuelle.
**Fix** : Migrer les filtres vers les composants shadcn/ui.

### H-5. CRM Dashboard : aucun etat vide explicite
**Page** : `src/app/(dashboard)/admin/crm/page.tsx`
**Probleme** : Si aucun prospect n'existe, les graphiques Recharts s'affichent vides sans
message explicatif. L'admin ne sait pas si c'est normal ou s'il y a un bug.
**Fix** : Ajouter un etat vide conditionnel "Aucun prospect. Creez votre premier prospect."

### H-6. Portail client : aucun toast d'erreur
**Page** : `src/app/(dashboard)/client/page.tsx` (L261-266)
**Probleme** : Le catch ne fait que `console.error`. Le client voit un spinner eternel
ou une page vide sans explication.
**Fix** : Ajouter un toast d'erreur et un etat explicite "Erreur de chargement".

### H-7. Trainers : pas de pagination
**Page** : `src/app/(dashboard)/admin/trainers/page.tsx`
**Probleme** : Tous les formateurs sont charges en une seule requete sans pagination.
Avec 100+ formateurs, la page deviendra lente.
**Fix** : Ajouter pagination Supabase + composant de pagination.

## MEDIUM - Agacant mais utilisable

### M-1. Incohérence de pagination entre pages
**Pages** : Comparaison apprenants/page.tsx vs apprenants/liste/page.tsx vs clients/page.tsx
**Probleme** : 3 implementations de pagination differentes :
- `clients/page.tsx` : utilise `Button` shadcn/ui avec Precedent/Suivant
- `apprenants/page.tsx` : pagination numerotee raw HTML avec style inline `#374151`
- `apprenants/liste/page.tsx` : autre pagination raw HTML avec boutons `<button>`
**Fix** : Creer un composant `Pagination` reutilisable.

### M-2. Apprenants liste : formulaire sans Zod
**Page** : `src/app/(dashboard)/admin/clients/apprenants/liste/page.tsx` (L150-173)
**Probleme** : Le formulaire "Ajouter un apprenant" utilise du state brut + validation manuelle
au lieu de React Hook Form + Zod (regle absolue CLAUDE.md #6). Pas de validation d'email.
**Concerne aussi** : Formulaire inline de clients/page.tsx, trainings/page.tsx (form de creation).
**Fix** : Migrer vers React Hook Form + Zod.

### M-3. Apprenants liste : `<select>` natif dans le dialog
**Page** : `src/app/(dashboard)/admin/clients/apprenants/liste/page.tsx` (L493-503)
**Probleme** : Le select "Entreprise" dans le dialog d'ajout est un `<select>` HTML natif
au lieu du composant `Select` shadcn/ui, creant une incoherence visuelle.
**Fix** : Remplacer par `<Select>` de shadcn/ui.

### M-4. Clients page : compteur affiche "clients.length" au lieu de "total"
**Page** : `src/app/(dashboard)/admin/clients/page.tsx` (L303)
**Probleme** : Le header affiche `{clients.length} entreprises` (nombre de la page courante)
au lieu de `{total}` (nombre total). Avec pagination, l'admin voit "10 entreprises" au lieu
de "47 entreprises".
**Fix** : Utiliser `{total}` au lieu de `{clients.length}`.

### M-5. Trainings page : responsive OK mais certains header overflow
**Page** : `src/app/(dashboard)/admin/trainings/page.tsx` (L374-414)
**Probleme** : Le header contient 4 elements alignes horizontalement (titre, view switcher,
CSV, bouton). Sur mobile, le flex ne wrap pas (`flex items-center gap-3`).
**Fix** : Ajouter `flex-wrap` ou reorganiser en 2 lignes sur mobile.

### M-6. Pas de chargement lazy des graphiques CRM
**Page** : `src/app/(dashboard)/admin/crm/page.tsx`
**Probleme** : Recharts est charge eagerly. Le bundle est alourdi meme si l'admin ne visite
jamais cette page.
**Fix** : `dynamic(() => import(...), { ssr: false })` pour les composants Recharts.

### M-7. Formulaire email masse sans validation de l'objet/corps
**Page** : `src/app/(dashboard)/admin/clients/apprenants/liste/page.tsx` (L565-567)
**Probleme** : Le bouton "Envoyer a tous" est desactive si subject/body vides, mais
il n'y a pas de validation de longueur min, de format, ni de preview avant envoi.
**Fix** : Ajouter un step de confirmation avec preview du message.

## LOW - Polish

### L-1. console.log/console.warn en production
**Pages multiples** : 28+ occurrences dans les pages de l'app.
**Exemples** : `documents/page.tsx:796`, `programs/import/page.tsx:177-200`,
`formations/[id]/_components/TabConventionDocs.tsx:443`.
**Fix** : Supprimer les console.log ou les conditionner a `process.env.NODE_ENV === 'development'`.

### L-2. TODO dans le code
**Fichiers** :
- `src/app/api/documents/attribute-secondary/route.ts:29` : TODO v2
- `src/app/(dashboard)/admin/formations/[id]/_components/TabConventionDocs.tsx:1590` : TODO extract helper
**Impact** : Faible, mais indique des fonctionnalites incompletes a verifier avant production.

### L-3. Sidebar : toutes les routes existent
**Fichier** : `src/components/layout/Sidebar.tsx`
**Statut** : Toutes les routes referencees dans la sidebar pointent vers des pages existantes.
Pas de lien mort detecte. Les routes masquees (Lieux, Certificateurs, Affacturage) sont
correctement commentees avec "Masque V1 -- Loris".

### L-4. Style inline "#374151" au lieu de variables CSS
**Pages** : `apprenants/liste/page.tsx`, `apprenants/page.tsx`, `trainers/page.tsx`
**Probleme** : Couleur entity hardcodee en inline style au lieu d'utiliser les variables
Tailwind ou le theme CSS. Ne fonctionnera pas pour C3V Formation (#2563EB).
**Fix** : Utiliser les classes Tailwind thematiques ou les variables CSS du theme.

### L-5. Formations : compteur de sessions total au lieu de filtre
**Page** : `src/app/(dashboard)/admin/trainings/page.tsx` (L377)
**Probleme** : Le badge affiche `{sessions.length}` (toutes les sessions) meme quand
des filtres sont actifs. Devrait afficher `{filtered.length}` / `{sessions.length}`.
**Fix** : `{filtered.length} / {sessions.length} sessions`.

### L-6. Client portal : "Documents" count hardcoded a 0
**Page** : `src/app/(dashboard)/client/page.tsx` (L322)
**Probleme** : La card "Documents" affiche `count: 0` en dur au lieu du vrai nombre.
**Fix** : Fetcher le count des documents du client.
