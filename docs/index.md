- [Onglet Questionnaires (Évaluation + Satisfaction)](./deep-dive-tab-questionnaires.md) — Analyse complète de `TabQuestionnaires` (395 LOC, vue unifiée 4 stages : avant/pendant/après/froid) + écosystème (23 fichiers, **~3 580 LOC**) : 2 composants UI, 14 routes API (7 PDFs + 3 batch-email + 2 tokens/submit public + 2 admin), 3 services, 3 templates HTML, **6 tables Supabase** (`questionnaires`, `questions`, `formation_evaluation_assignments`, `formation_satisfaction_assignments`, `questionnaire_responses`, `questionnaire_tokens`), 5 migrations. État des lieux **sévère : fonctionnel à ~40 % avec 5 bugs P0 bloquants en production** : (1) **découplage architectural** — TabQuestionnaires écrit dans `formation_evaluation_assignments` mais TOUS les consommateurs (PDFs, KPIs Qualiopi via `loadQualiopiIndicators`, portail learner, auto-send cron) lisent `questionnaire_sessions` → admin attribue dans le vide, (2) **RLS permissive** `FOR ALL USING (entity match)` sur les 3 nouvelles tables → learner peut INSERT/DELETE, (3) **enum incompatible** `satisfaction_entreprise` proposé par UI mais CHECK constraint DB le refuse, (4) **bug `Boolean("non") === Boolean("oui")` = true** dans `load-evaluation-results.ts:59` → toutes les questions yes_no donnent 100 % de score, (5) automatisations Qualiopi standard (J-3 / J0 / J+7 / J+30) envoient des emails sans pièce jointe ni lien. Score qualité **3/10 vs 8/10 pour TabConventionDocs post-solidification**. Plan : 6 volets A-F, **65h dont 16-24h critiques sur le refacto architectural (Volet E)**. Généré le 2026-05-25.
