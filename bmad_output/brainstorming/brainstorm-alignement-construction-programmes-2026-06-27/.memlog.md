- (direction by user) Vision : la plateforme DEVRAIT faire ce travail (contenu + mise en page) a la place de ChatGPT+Gamma.
- (insight by user) Gamma n'apporte QUE 'un PDF propre sans retouche' : il prend la sortie telle quelle, aucune retouche manuelle. => pas besoin d'editeur de mise en page interactif ; un template PDF propre suffit a remplacer Gamma.
- (decision by coach) Gamma remplacable a 100% par un template PDF interne si la qualite visuelle est au niveau, puisque zero retouche cote client.
- (decision by user) Les 2 PDF fournis (vert 'Bien installer le resident' + bleu 'Communication manageriale N2') = le niveau de propre cible. Si la plateforme sort exactement ca, il n'ouvre plus jamais Gamma.
- (decision by user) Un seul modele/habillage standard 4 pages suffit, pas de choix de couleur ni de retouche.
- (insight by user) Le 1er jet de ChatGPT est generalement le bon -> 'un clic -> PDF pret' est viable, pas besoin de gros editeur.
- (decision by user) Flux retenu : generer -> relecture rapide a l'ecran -> bouton Regenerer + correction manuelle legere possible -> Generer le PDF. Filet de securite leger, pas d'editeur lourd.
- (decision by user) UN SEUL chemin de creation : l'IA remplace la construction manuelle sequence-par-sequence. MAIS conserver telle quelle la logique de lien programme -> session existante (le modele de donnees programme/sessions ne change pas, seule la facon de creer le contenu change).
- (decision by user) Inputs de generation : nom + duree pre-remplis automatiquement depuis la formation + un champ libre optionnel 'precisions' pour cas particuliers (ex : DPC/TP, public cible specifique).
- (decision by coach) Le prompt de generation = le prompt-type fourni par le client, parametre par nom/duree/precisions (a confirmer comme system prompt).
- (insight by user) Generation aussi EN AMONT cote CRM/prospect : le programme est d'abord un DOCUMENT DE VENTE (avant signature), pas seulement un programme pedagogique. Le bouton doit exister cote CRM/prospect, pas que dans l'onglet Programme d'une formation.
- (insight by coach) Meme moteur sert 2 moments : proposition prospect (vente) PUIS programme de la formation (signe). Le 'programme' est fondamentalement une activite commerciale d'abord.
- (decision by user) PAS de report automatique prospect -> formation. Raisons (les 3) : le programme change entre la proposition et la formation reelle ; un prospect peut donner plusieurs formations (pas du 1:1) ; le lien prospect<->formation n'est pas propre. => le generateur est un OUTIL dispo aux 2 endroits ; toute reprise du programme prospect vers la formation reste manuelle/optionnelle, jamais automatique.
- (insight) SYNTHESE / direction retenue : le generateur de programme est un OUTIL AUTONOME REUTILISABLE (genere par IA avec le prompt client + template PDF 4 pages interne = qualite des 2 exemples, Gamma supprime), pose a 2 endroits (CRM/prospect comme doc de vente, ET onglet Programme d'une formation). Flux quasi un-clic : nom+duree auto + champ precisions -> generation -> relecture rapide / regenerer / corriger -> PDF. Remplace la saisie manuelle sequence-par-sequence (un seul chemin) tout en conservant le lien programme->session existant. Pas de report auto prospect->formation (reutilisation manuelle). BREAKTHROUGH : la construction de programme est d'abord un ACTE COMMERCIAL (doc de vente), pas pedagogique -> c'est ce qui justifie sa place dans le CRM et dissout le faux probleme de 'doublon'.
- (insight) MODELE DE DONNEES REEL : une session n'est jamais reliee directement a un programme. Chaine = session.training_id -> training.program_id -> programs. Le point d'accrochage du programme = la FORMATION (training), pas la session ; les sessions heritent du programme via leur training automatiquement. Consequence scenario prospect : on rattache le programme au moment de creer la formation (set training.program_id), geste manuel volontaire ; les sessions suivent seules.
- (question) DECISION A TRANCHER EN SPEC : programme genere cote prospect = (A) simple PDF de vente puis regeneration a la signature, OU (B) vraie ligne programs 'flottante' rattachee ensuite a la formation d'un clic (anti-doublon, mais necessite un stockage programme<->prospect qui n'existe pas aujourd'hui).
- (decision by user) TRANCHE = option B : a la generation cote prospect, sauver une vraie ligne programs 'flottante' (+ PDF de vente), puis la rattacher a la formation d'un clic au moment de la signature (set training.program_id).
